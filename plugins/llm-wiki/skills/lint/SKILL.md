# Lint 루틴

ONDA product-docs 위키의 건강 상태를 체계적으로 점검하고 결과를 보고한다.

## Phase 0: product-docs 경로 해석 및 cwd 전환

이 스킬은 `product-docs` 리포 루트를 기준으로 동작한다. 호출 시 cwd 가 product-docs 가 아닐 수 있으므로 다음 순서로 해석한 뒤 `cd` 한다:

1. `$PRODUCT_DOCS_PATH` 환경변수
2. `$TPORT_HOME` 환경변수 → `$TPORT_HOME/product-docs`
3. 후보 경로 순차 확인:
   - `~/Workspaces/tport/product-docs`
   - `~/dev/tport/product-docs`
   - `~/code/tport/product-docs`
   - `~/projects/tport/product-docs`
4. 모두 실패 시 사용자에게 경로 문의 후 중단

해석된 경로에 `cd` 한 뒤 `git pull origin main --ff-only` 로 최신화.

## 트리거 조건

- "린트 돌려줘", "문서 점검해줘", "깨진 링크 확인해줘"
- "고아 문서 찾아줘", "문서 건강 점검", "INDEX 일관성 확인"
- 대규모 인제스트 작업 후 자동 제안 가능

## 인자

- **범위**: 전체(`all`), 특정 프로젝트(`projects/[이름]`), 특정 제품(`products/[이름]`)
- 인자가 없으면 전체 저장소 대상

예시:
- `/lint` → 전체 점검
- `/lint projects/2026-01-ONDA-CMS-더휴식-Edition` → 해당 프로젝트만
- `/lint products/OSP` → 해당 제품만

## 점검 항목

### 1. 깨진 링크

마크다운 파일 내 `[text](path.md)` 형태의 링크가 실제 파일을 가리키는지 확인.

**방법:**
1. Grep 으로 마크다운 링크 패턴 검색: `\[.*\]\(.*\.md\)`
2. 각 링크의 대상 파일이 실제로 존재하는지 Glob/Read 로 확인
3. 외부 URL(`http://`, `https://`) 은 검사 대상에서 제외

**보고 형식:**
```
| 파일 | 깨진 링크 | 수정 제안 |
|---|---|---|
```

### 2. 고아 문서

어떤 문서에서도 링크되지 않는 파일을 찾는다.

**방법:**
1. Glob 으로 대상 범위의 모든 `.md` 파일 목록 수집
2. 모든 파일 내 링크 참조 목록 수집
3. 참조되지 않는 문서 식별
4. 제외 대상: `INDEX.md`, `README.md`, `CHANGELOG-*.md`, `CLAUDE.md`, `LOG.md`, `_workspace/` 내 파일

**보고 형식:**
```
| 고아 문서 | 권장 조치 |
|---|---|
```

### 3. INDEX 일관성

`INDEX.md` 와 실제 폴더 구조가 일치하는지 확인.

**방법:**
1. 루트 `INDEX.md` 에 나열된 프로젝트/제품 목록 파싱
2. `projects/`, `products/` 실제 폴더 목록과 비교
3. INDEX 에는 있지만 폴더가 없는 항목 식별
4. 폴더는 있지만 INDEX 에 없는 항목 식별
5. `products/INDEX.md` 도 동일하게 검사

**보고 형식:**
```
| 불일치 유형 | 항목 | 위치 |
|---|---|---|
| INDEX 에만 존재 | ... | ... |
| 폴더에만 존재 | ... | ... |
```

### 4. 프로젝트-제품 양방향 링크

프로젝트 README 의 `관련 제품` 과 제품 README 의 `관련 프로젝트` 가 양방향으로 일치하는지 확인.

**방법:**
1. 각 프로젝트 README 에서 `관련 제품` 필드 파싱
2. 해당 제품 README 에서 `관련 프로젝트` 섹션에 해당 프로젝트가 있는지 확인
3. 역방향도 동일하게 검사

### 5. 누락된 교차 참조

관련성이 높지만 서로 링크가 없는 문서 쌍을 식별.

**방법:**
- 같은 프로젝트 내 문서 간 교차 참조 확인
- 동일 엔티티/키워드를 언급하지만 링크가 없는 문서 식별

### 6. 상충하는 내용 (선택적)

사용자가 요청한 경우에만 수행. 문서 간 모순되는 정보를 찾는다.

- 동일 주제에 대해 다른 수치, 날짜, 사양을 기술하는 문서
- 상태가 다르게 기록된 항목

### 7. 누락된 페이지

여러 문서에서 언급되지만 전용 페이지가 없는 개념이나 엔티티를 식별.

## 실행 후 처리

### 결과 보고

점검 완료 후 사용자에게 요약 보고서 제공:

```markdown
## 린트 결과 요약

| 점검 항목 | 결과 |
|---|---|
| 깨진 링크 | N건 |
| 고아 문서 | N건 |
| INDEX 불일치 | N건 |
| 양방향 링크 누락 | N건 |
| 누락된 교차 참조 | N건 |
| 상충 내용 | N건 (수행 시) |
```

### 자동 수정 제안

발견된 문제에 대해:
- **자동 수정 가능**: 깨진 링크(경로 오타), INDEX 누락 → 사용자 승인 후 즉시 수정
- **수동 판단 필요**: 고아 문서 처리 방향, 상충 내용 해소 → 사용자에게 판단 요청

### LOG.md 기록

린트 수행 후 `LOG.md` 상단에 결과 기록:

```markdown
## [YYYY-MM-DD] lint | [범위] 문서 건강 점검

- 깨진 링크 N건 수정: [경로1], [경로2]
- 고아 문서 N건 발견: [경로] → [조치]
- INDEX 불일치 N건 수정
- 상충 내용 N건
```

## 주의사항

- `_workspace/` 내 파일은 린트 대상에서 제외 (개인 작업 공간)
- `raw/` 폴더 내 파일은 원천 자료이므로 수정하지 않음
- 자동 수정 전 반드시 사용자 승인 요청
- 수정 사항 commit/PR 은 사용자 확인 후 별도 진행 (이 스킬은 자동 commit 하지 않음)
